整体架构方案对比

基于已确认的技术栈（Streamlit + LangChain ReAct + OpenAI），三种架构组织方式

A

单 Agent + 多 Tool 模式（推荐）

一个 ReAct Agent 配备多个工具，由 Agent 自主决定调用哪个工具、调用几次。

架构示意
用户输入（自然语言）
↓
🤖 ReAct Agent (GPT-4o)
意图理解 → 选择工具 → 执行 → 观察 → 回答
↓ 调用工具（可多次）
🔍 query_metric
预定义查询
🐍 run_pandas
复杂代码查询
📊 visualize
Plotly图表
🔗 analyze_cause
因果分析

优势

  • 架构最简洁，一个Agent统一管理
  • Agent可自主组合多个工具完成复杂任务
  • ReAct模式天然支持思考→行动→观察循环
  • LangChain原生支持，代码量最少

劣势

  • 复杂问题可能需要多轮工具调用，延迟略高
  • 所有逻辑集中在一个Agent的Prompt中
B

路由 Agent + 子 Agent 模式

一个路由Agent判断问题类型，然后分发给专门的子Agent处理。每个子Agent专精一个领域。

架构示意
用户输入
↓
🔀 路由 Agent
分类 → 分发
📊 查询Agent
数据+图表
🔗 因果Agent
分析归因
💬 对话Agent
闲聊引导

优势

  • 每个子Agent职责清晰
  • 可独立优化每个领域

劣势

  • 架构复杂，LLM调用次数翻倍
  • 跨Agent共享上下文困难
  • 过度工程化，小项目不必要
C

意图分类 + 直接执行模式

先用LLM做意图分类，然后直接调用对应的处理函数，不走Agent循环。

架构示意
用户输入
↓ LLM分类
查询意图
→ query_fn()
可视化意图
→ viz_fn()
因果意图
→ causal_fn()
↓ 结果 → LLM生成回答

优势

  • 最快，只需2次LLM调用
  • 最可预测，无循环风险

劣势

  • 无法处理需要多步推理的问题
  • 不算真正的Agent，P2加分项弱
  • 无法自主组合工具
