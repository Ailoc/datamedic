# DataMedic 现代会话套件 UI 设计

## 背景

当前项目是医院运营指标智能分析助手，前端由 `src/datamedic/app.py` 使用 Streamlit 构建，后端由 FastAPI 提供 `/chat`、语音识别和语音合成能力。现状中，`app.py` 同时承担页面渲染、侧边栏提示、消息状态、API 调用和语音组件嵌入，界面简约但缺少现代产品感；侧边栏目前是使用提示，不能管理会话。

本次设计确认采用「会话套件」方向：整体更像一个温暖、可信赖的医院运营数据顾问，而不是传统冷硬后台。界面需要保持专业信息密度，同时具有现代、优雅、可弹性伸缩的视觉表现。会话需要保留在本地文件中，刷新页面或重启 Streamlit 后仍然可恢复。

## 目标

1. 将 Streamlit 前端重设计为现代、优雅、温暖的顾问式聊天界面。
2. 将侧边栏从「使用提示」改为真正的会话侧边栏，支持新建、切换、删除会话。
3. 使用本地 JSON 文件持久化会话元数据和前端消息历史。
4. 保持后端 Agent 的 `session_id` / `thread_id` 隔离机制，确保多会话上下文互不污染。
5. 对前端状态和 API 调用逻辑做适度拆分，让代码更清晰、可测试、可维护。
6. 页面设计必须弹性伸缩，在常见桌面宽度和较窄窗口下都保持稳定布局，不出现文字溢出、组件重叠或主内容不可用。
7. 左上角的软件图标和名称必须有设计感，成为首屏清晰的品牌识别点，而不是普通文本标题。

## 非目标

1. 不引入数据库、用户账号、云端同步或多用户权限系统。
2. 不改变现有数据分析工具、指标口径、Agent 提示词和图表生成能力。
3. 不重写为 React/Vue/Dash 等其他前端框架。
4. 不把会话 CRUD 搬到后端 API，除非后续需要多客户端共享会话。
5. 不在本轮实现全文搜索、会话重命名弹窗或复杂标签体系。

## 设计方向

视觉方向采用「会话套件」：

- 色彩：温暖白、柔和米白、深医疗绿、少量深墨色作为主要文字和品牌色。避免单一色相铺满页面，整体要干净、安静、专业。
- 质感：轻边框、微阴影、紧凑圆角，避免夸张卡片堆叠和营销式大英雄区。
- 信息结构：左侧是会话管理，右侧是当前会话主工作区，底部是稳定输入区。
- 语气：文案简洁，让用户直接进入问数，不使用大段功能说明占据页面。

左上角品牌区设计为一个小型产品标识组件：

- 图标使用纯 CSS 构建的医疗数据符号，例如深绿圆角方块内含脉冲线、柱状图或十字与数据点的组合。
- 名称使用 `DataMedic` 或当前中文产品名的紧凑组合，主标题加粗，副标题为「医院运营数据顾问」。
- 品牌区固定在会话侧边栏顶部，在窄屏时仍保持可辨识，必要时隐藏副标题但保留图标和名称。

## 页面结构

### 会话侧边栏

侧边栏替换原有使用提示，承担以下职责：

- 展示品牌图标和产品名称。
- 提供「新建会话」主按钮。
- 展示最近会话列表。
- 高亮当前会话。
- 每个会话显示标题、最后更新时间和最近一条消息摘要。
- 支持删除会话，并在删除当前会话后自动切换到剩余最近会话；如果没有剩余会话，则自动创建新会话。
- 在底部用小型数据概览展示数据范围、科室数、指标数，不再占用主内容空间。

会话标题自动生成：

- 新建会话默认标题为「新的运营问答」。
- 用户发送第一条消息后，用消息前 18 到 24 个中文字符生成标题。
- 后续不自动覆盖用户已形成的标题，避免会话列表跳动。

### 主工作区

主工作区由三部分组成：

- 顶部会话标题栏：展示当前会话标题、数据范围胶囊、当前状态。
- 中部聊天流：渲染用户消息、助手消息、Plotly 图表和错误提示。
- 底部输入区：包含语音入口、文本输入和发送动作，视觉上是一体化输入条。

空会话状态不使用说明书式文本，而提供少量可点击示例问题：

- 「展示 2025 年骨科出院人次趋势」
- 「心内科和心外科手术人次对比」
- 「哪个科室手术人次最多」
- 「为什么门诊人次下降」

### 响应式与弹性伸缩

Streamlit 仍然运行在桌面 Web 页面中，因此主要优化桌面和窄桌面窗口：

- 使用宽屏布局，主体容器最大宽度自适应窗口。
- 侧边栏采用固定视觉宽度但允许 Streamlit 原生折叠。
- 主聊天区域使用 `min-width: 0`、合理的 `max-width` 和换行规则，长中文、英文指标名和错误信息不会撑破布局。
- 图表使用 `use_container_width=True`，外层容器宽度跟随主内容区变化。
- 按钮、会话项和输入条设置稳定高度，hover、选中、错误状态不改变布局尺寸。
- 在较窄窗口下，减少顶部状态胶囊数量或让它们自然换行；聊天气泡宽度使用百分比和最大宽度约束。

## 数据与状态设计

新增前端会话仓库，建议文件为 `src/datamedic/ui/conversation_store.py`。

核心数据结构：

```python
{
    "active_session_id": "uuid",
    "sessions": [
        {
            "id": "uuid",
            "title": "胸外科门诊趋势",
            "created_at": "2026-05-13T00:00:00+08:00",
            "updated_at": "2026-05-13T00:05:00+08:00",
            "messages": [
                {
                    "role": "user",
                    "text": "展示2025年胸外科门诊人次趋势",
                    "figures": []
                },
                {
                    "role": "assistant",
                    "text": "以下是趋势分析...",
                    "figures": [{ "...": "plotly figure json" }]
                }
            ]
        }
    ]
}
```

存储位置：

- 默认使用项目内可忽略目录，例如 `.datamedic/conversations.json`。
- 目录不存在时自动创建。
- JSON 损坏时不让页面崩溃，自动备份损坏文件为带时间戳的 `.bak`，并初始化空会话状态。

会话仓库职责：

- `load_state()`：读取本地 JSON 并返回规范化状态。
- `save_state(state)`：原子写入状态，降低写坏文件风险。
- `ensure_active_session(state)`：保证总有一个可用当前会话。
- `create_session(state)`：创建新会话并设为当前会话。
- `delete_session(state, session_id)`：删除会话并选择新的当前会话。
- `set_active_session(state, session_id)`：切换当前会话。
- `append_message(state, session_id, message)`：追加消息、更新时间并在首条用户消息后生成标题。

`st.session_state` 只缓存当前渲染所需状态，不作为唯一事实来源。每次会话操作后写回本地 JSON。

## 后端与 API 设计

后端 `/chat` 请求结构保持不变：

```json
{
  "session_id": "uuid",
  "message": "用户问题"
}
```

FastAPI 继续将 `session_id` 传给 LangGraph：

```python
config={"configurable": {"thread_id": request.session_id}}
```

这样每个前端会话天然对应独立 Agent 线程。删除前端本地会话时，本轮不强制清理后端内存检查点，因为当前后端使用进程内 Agent 对象和 thread 配置，且没有持久数据库；删除语义先定义为「从前端会话列表和本地历史中移除」。后续若引入持久 LangGraph checkpointer，再补充后端清理接口。

后端代码需要轻量重构：

- 将图表提取逻辑从路由函数中拆为私有 helper，便于测试。
- 保留 `ChatRequest` / `ChatResponse` 兼容性。
- 对空输出、工具异常、请求异常保持友好错误文本。

## 前端代码组织

建议拆分为以下模块：

- `src/datamedic/app.py`：Streamlit 入口，负责页面组装和事件流。
- `src/datamedic/ui/styles.py`：集中输出 CSS，包含品牌、侧边栏、聊天气泡、输入区、响应式规则。
- `src/datamedic/ui/conversation_store.py`：本地会话 JSON 仓库。
- `src/datamedic/ui/chat_client.py`：封装 `/chat` HTTP 请求和错误处理。
- `src/datamedic/ui/__init__.py`：UI 辅助包导出。

`app.py` 保留 Streamlit 的声明式结构，但避免继续堆积所有细节。推荐渲染流程：

1. 初始化页面配置和 CSS。
2. 加载会话状态，确保存在当前会话。
3. 渲染会话侧边栏，并处理新建、切换、删除事件。
4. 渲染主工作区头部。
5. 渲染当前会话消息和图表。
6. 渲染语音组件。
7. 处理 `st.chat_input` 提交，追加用户消息、调用后端、追加助手回复并保存。

## 错误处理

前端错误处理：

- 后端连接失败：展示「无法连接到后端服务，请确认 FastAPI 已启动」。
- 请求超时：展示「分析时间较长，请稍后重试或缩小问题范围」。
- 响应 JSON 异常：展示「后端返回格式异常」。
- Plotly JSON 解析失败：展示文字回复，并在图表位置显示轻量错误提示。
- 本地会话文件损坏：自动备份并初始化，不阻断页面加载。

后端错误处理：

- `/chat` 保持返回 `ChatResponse`，避免前端出现未处理异常页面。
- 图表工具提取失败不影响文字回答。
- Agent 抛错时返回友好中文说明。

## 测试策略

优先补充不依赖真实 LLM 的单元测试：

- 会话仓库创建默认状态。
- 创建会话会设为 active。
- 删除当前会话后自动切换到其他会话。
- 删除最后一个会话后自动创建新会话。
- 追加首条用户消息后生成标题。
- JSON 损坏时备份并恢复空状态。
- Chat client 在连接错误、超时、非 200 响应时返回友好错误对象。
- 后端图表提取 helper 在没有工具消息时返回空列表。

UI 视觉验证：

- 启动 FastAPI 和 Streamlit。
- 在浏览器中检查会话新建、切换、删除、刷新后恢复。
- 检查较宽桌面和较窄窗口下布局不重叠、不溢出。
- 检查左上角品牌图标和名称在侧边栏展开状态下清晰，有设计感。
- 检查图表随容器宽度变化。

## 验收标准

1. 侧边栏不再显示原来的大段使用提示，而是会话管理界面。
2. 用户可以新建、切换、删除会话。
3. 会话历史保存到本地 JSON，刷新页面或重启 Streamlit 后仍可恢复。
4. 每个会话使用独立 `session_id` 调用后端，保持上下文隔离。
5. 页面采用「会话套件」视觉方向，整体现代、温暖、优雅。
6. 左上角品牌图标和软件名称有明确设计，不是普通标题文本。
7. 页面弹性伸缩良好，窄窗口下无明显溢出、重叠或不可操作区域。
8. 现有聊天、Plotly 图表渲染和语音组件能力继续可用。
9. 新增和重构代码有单元测试覆盖核心状态逻辑。
10. 全量测试通过。

## 实施顺序

1. 为本地会话仓库写失败测试，再实现仓库。
2. 为 chat client 错误处理写失败测试，再实现封装。
3. 提取后端图表 helper 并补充测试。
4. 添加集中 CSS 和品牌组件样式。
5. 重构 `app.py` 为会话侧边栏与主聊天工作区。
6. 手动启动应用做浏览器视觉和交互验证。

## 自检

- 没有未定需求或占位项。
- 设计已明确采用「会话套件」方向。
- 会话持久化明确为本地 JSON 文件。
- 后端清理语义已说明：本轮删除前端本地历史，不清理非持久 LangGraph 线程。
- 弹性伸缩和品牌设计要求已纳入目标与验收标准。
